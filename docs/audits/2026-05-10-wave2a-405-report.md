## Hardening pass — 2026-05-10 (post-review by RRSS coordinator)

### Trigger

Initial integration of `PublishFlags` into `publish_guard` only emitted
a `publish_guard.dry_run` marker without blocking. A future caller that
ignored ops_log could still proceed to a real publish even when the
runtime flags forbade it. The stop button was passive, not active.

### Decision

Promote the flag check to a real block:

* Option A chosen — same `assert_can_publish` function raises
  `PublishBlockedError` when `flags` is explicit AND
  `flags.allows_real_publish() is False`. Reasons drawn from
  `PublishFlags.block_reasons()`. Ops_log event renamed
  `publish_guard.dry_run` → `publish_guard.runtime_block`.
* Option B (separate `assert_runtime_allows_publish`) was rejected to
  avoid two code paths a caller could forget to invoke.

Backward compatibility (`flags=None` → byte-identical to pre-#405) is
preserved and covered by `test_call_without_flags_preserves_legacy_behavior`.

### Changes applied (delta from initial PR #407)

* `scripts/discovery/lib/publish_flags.py`:
  * New `PublishFlags.block_reasons() -> list[str]` returning the ordered
    subset of `RUNTIME_BLOCK_REASONS = (publish_disabled, dry_run_enabled,
    max_posts_zero)`.
  * New `PublishFlags.cross_validation_warnings() -> list[str]` returning
    a stable subset of `CROSS_VALIDATION_CODES = (publish_with_dry_run,
    publish_with_zero_cap, daily_cap_below_per_run, daily_cap_not_enforced)`.
  * `from_env` emits a WARNING per cross-validation code via
    `logging.getLogger("scripts.discovery.lib.publish_flags")`.
  * Module docstring promoted `MAX_POSTS_PER_DAY` to "parsed but not
    enforced until #404-lite".
* `scripts/discovery/lib/publish_guard.py`:
  * `assert_can_publish` checks `flags.allows_real_publish()` BEFORE
    evaluating gates or touching the DB.
  * On runtime block: emits `publish_guard.runtime_block` (with the
    parsed flags echoed and `cross_validation` list) and raises
    `PublishBlockedError(flags.block_reasons(), ...)`.
  * `flags=None` path unchanged (verified by the legacy test).
* `tests/lib/test_publish_flags.py`:
  * 11 new tests covering `block_reasons` (empty / default / each
    individual reason / all combined / subset of `RUNTIME_BLOCK_REASONS`)
    and `cross_validation_warnings` (each code emitted on the right
    combination, `daily_cap_not_enforced` always present, `from_env`
    emits the warnings via logging, clean config emits ONLY the
    informational warning).
* `tests/discovery/test_publish_guard_flags_integration.py`:
  * Replaced 4 dry-run-marker tests with 4 blocking tests:
    `failclosed_flags_block_real_publish_with_runtime_block_event`,
    `default_env_is_failclosed_and_raises`,
    `max_posts_zero_is_a_runtime_block_reason`,
    `allon_flags_pass_through_to_gates`. Plus the legacy back-compat
    test unchanged.
* `docs/runbooks/publish-emergency-stop.md`:
  * Added Step 0 (page David before touching anything).
  * Added Step 1.5 (kill in-flight publisher processes via `pkill -f`,
    stop user systemd units, with objective stop verification echoing
    `OK` / `FAIL`).
  * Added Step 1.6 (revert deployment via clean revert PR — never
    `git reset --hard` on `main` from the VPS).
  * Added paste-and-run verification 3-5: `pgrep` no publisher,
    `systemctl --user` no active unit, end-to-end `publish_guard` raise
    smoke from `os.environ`.
* `docs/editorial-pipeline/runtime-flags-contract.md`:
  * Documented `block_reasons` and `cross_validation_warnings` API.
  * Added "Stop-button semantics" section detailing the three modes
    (`flags=None`, `flags` allowing publish, `flags` blocking publish).
  * Promoted `MAX_POSTS_PER_DAY is NOT enforced in #405` from a footnote
    to its own section.
