# Wave 2.A — Publishing readiness plan

**Date**: 2026-05-10
**Status**: planned
**Predecessor**: PR #400 squash `d835122` (Wave 1.5 merged 2026-05-10)
**Successor**: Wave 2.B (operational hardening) → Wave 2.C (multiformato)

## Goal

> Leave the system capable of executing a controlled first publish, **without executing it yet**.

Wave 2.A does NOT activate real publication. It builds the safety scaffolding (hash final, kill switch, observability log, governance docs) so the next milestone can publish a single controlled item with full traceability.

## Scope (4 deliverables)

| # | Issue | Title | Type | Blocking? |
|---|---|---|---|---|
| 1 | #402 | publication_content_hash + E2E dry-run test | code | yes |
| 2 | #405 | Stop button + runtime flags + emergency runbook | code+doc | yes |
| 3 | #404-lite | Structured publish_guard log (no dashboard) | code | yes |
| 4 | #406 | Source-use policy + secrets/tokens runbook | doc | non-blocking, parallel |

## Order of execution

1. **#402** first (publication_content_hash). Blocks everything downstream.
2. **#405** second (runtime flags + stop button). Required before any code path can attempt publish.
3. **#404-lite** third (structured log). Required to operate first publish with traceability.
4. **#406** in parallel with any of the above (pure documentation).

## Out of scope (Wave 2.A)

- Real publication to any platform (blog, LinkedIn, newsletter).
- Cron activation.
- `variants.py` refactor (#401, deferred to Wave 2.C).
- Full SQLite hardening (#403, deferred to Wave 2.B).
- Visual dashboard (deferred to Wave 2.B).
- Multiformato (carousel/video, deferred to Wave 2.C).

## Definition of Done (per issue)

Every Wave 2.A issue must satisfy all 8 points before merge:

1. Contract written before code (interface + behavior doc).
2. Unit tests.
3. Local integration test.
4. VPS smoke run.
5. "Repo dice X / VPS muestra Y" verification report.
6. No real writes unless issue explicitly authorizes (Wave 2.A authorizes none).
7. Rollback or stop path documented.
8. Explicit "what this does NOT solve".

## Restrictions in force throughout Wave 2.A

- No publication activation.
- No cron.
- Stage 7.5 frozen.
- No LinkedIn token activation.
- Blog (`umbralbim.cl`, part of `umbral-bot-2`) is the **future first canal**, but NOT yet wired to the pipeline. Treat as target, not as available.

## First publish milestone (Milestone 2 — out of Wave 2.A scope)

When Wave 2.A closes (all 4 deliverables merged), the next milestone will be a **single controlled publish** with this preferred route:

1. Dry-run real with real data.
2. Blog draft on `umbralbim.cl` (within `umbral-bot-2`).
3. Human review (David).
4. Controlled publication with `PUBLISH_ENABLED=true MAX_POSTS=1` for that single execution.
5. LinkedIn deferred to a later milestone.

This milestone requires #406 merged (source-use policy + secrets runbook) and explicit authorization from David.
