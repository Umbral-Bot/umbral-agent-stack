## Addendum D-047-bis-C — wiring opt-in (2026-05-14, segundo commit del PR)

### Decisión

David eligió opción C: wirear el módulo en `aeco-kb-pipeline.bicep` con un flag apagado por default, sin desplegar nada.

### Hallazgo previo a aplicar

Al re-leer `infra/azure/aeco-kb-pipeline.bicep` para aplicar el cambio se descubrió que **el módulo ya estaba wireado** (líneas ~73-86, junto a `sourceCrawler` e `indexPipeline`). El audit original reportó R2 incorrectamente: el grep inicial buscó `pdf.parser|aeco-pdf-parser` sobre el ARM JSON compilado, pero el módulo se referencia como identifier `pdfParser` (camelCase) y el path con guiones sale en otra línea del JSON — el match no se capturó en la salida truncada del grep.

Gap real: el módulo se desplegaba **siempre que se invocara el umbrella**. No había opt-in, lo que significaba que cualquier `az deployment group create` sobre `aeco-kb-pipeline.bicep` materializaba el job aun sin imagen GHCR pusheada todavía. Eso es exactamente lo que David quiso evitar al pedir el flag.

### Cambios aplicados

Archivo modificado: `infra/azure/aeco-kb-pipeline.bicep`.

1. Nuevo parámetro:
   ```bicep
   @description('Si true, despliega el ACA Job aeco-pdf-parser. Default false: el módulo está cableado y compilable, pero no se materializa en Azure salvo opt-in explícito (D-047-bis-C, 2026-05-14).')
   param deployPdfParser bool = false
   ```
2. Module reference convertido a condicional:
   ```bicep
   module pdfParser 'modules/aeco-pdf-parser-job.bicep' = if (deployPdfParser) { ... }
   ```
3. Output del nombre del job hecho condicional con non-null assertion para evitar BCP318:
   ```bicep
   output parserJobName string = deployPdfParser ? pdfParser!.outputs.name : ''
   ```
4. Output adicional para diagnóstico:
   ```bicep
   output pdfParserDeployed bool = deployPdfParser
   ```

### Validaciones del addendum

- `az bicep build infra/azure/aeco-kb-pipeline.bicep` → **EXIT 0, 25396 bytes ARM, 0 warnings, 0 errors** (la 1ª pasada generó BCP318 sobre el output ternario; corregido con `pdfParser!.outputs.name`).
- `az bicep build infra/azure/main.bicep` → **EXIT 0, 68258 bytes** (sin cambios en main; main no referencia el pipeline directamente).
- `pytest tests/test_pdf_parser.py` → **16 passed in 0.86s** (no regresión; tests son puro Python, no afectados por Bicep).
- Confirmación en ARM compilado: la entrada `"condition": "[parameters('deployPdfParser')]"` aparece sobre el deployment del módulo `aeco-pdf-parser-job`. Default value confirmado en `false`.

### Comportamiento por default

Un `az deployment group create -f aeco-kb-pipeline.bicep -p ...` sin pasar `deployPdfParser=true` **NO crea ni modifica el ACA Job pdf-parser**. El crawler y el publisher sí se despliegan (no fueron tocados, conservan el comportamiento previo). Si después de smoke 044+048+049 se quiere activar 047, basta con agregar `-p deployPdfParser=true`.
