## Hallazgo principal

**La IP de la VPS está bloqueada por YouTube en los 3 endpoints**:

- **alt1 yt-dlp**: 15/15 fallan con `Sign in to confirm you're not a bot. Use --cookies-from-browser or --cookies` (anti-bot por IP de datacenter, problema conocido y documentado en yt-dlp GitHub issues).
- **alt2 youtube-transcript-api**: 15/15 fallan con `RequestBlocked: YouTube is blocking requests from your IP`. Confirma bloqueo a nivel IP (no API key, no cookies — solo HTTP headers).
- **alt3 atom feed**: 2/15 OK en una corrida (canal `UCWstLaT61QUc-TvfxOjNpFw` para 2 videos del mismo referente). Los otros 13 devuelven 404. Una probe manual posterior contra los mismos channel_ids (incluido el que sí funcionó) devuelve 404 para todos — sugiere **rate limiting** del propio feed XML cuando se hacen ráfagas desde la misma IP.

Adicional: el regex `"channelId":"(UC...)"` aplicado al watch HTML matchea la primera ocurrencia, que en muchas páginas es un canal recomendado del sidebar y no el del video. Para usar alt3 en serio habría que parsear el JSON `ytInitialPlayerResponse.videoDetails.channelId` del HTML, no un regex sobre el bloque entero.

## Recomendación

**Ninguna alternativa supera el umbral del 75% desde la VPS tal como está hoy.** No tiene sentido implementar el adapter Stage 2 (013-K) hasta resolver el bloqueo de IP. Opciones, en orden de fricción creciente:

1. **`cookies-from-browser` o `--cookies` con sesión autenticada** (yt-dlp): copiar cookies de un browser logueado en YouTube en una máquina residencial. Bajo costo, requiere refresh periódico de cookies, y técnicamente viola TOS.
2. **Proxy residencial** (BrightData / Oxylabs / similar): unblockea las 3 alternativas con altísima fiabilidad. Costo recurrente (~USD 15-50/mes según volumen).
3. **Ejecutar la extracción desde el Worker Windows VM** (residencial, no datacenter): si la VM tiene IP residencial, alt1/alt2 deberían funcionar sin cookies. Reusa infra existente.
4. **YouTube Data API v3** (oficial): requiere API key, cuota diaria de 10k units. Da `snippet.description`, `snippet.tags`, `contentDetails.duration`, `statistics`, pero **no** transcripts ni descripciones largas más allá de las que el creador haya escrito. Más fiable a largo plazo, pero no resuelve el problema de transcripts.
5. **Aceptar `created_no_body` para `canal=youtube`** y diferir el contenido a una segunda fase manual o a un worker diferente. Costo cero pero degrada la utilidad para David.

**Mi recomendación operativa**: combinar (3) — mover la extracción YouTube al Worker Windows VM mientras esté disponible — con (4) como fallback para datos estructurados (duration, view_count, tags). Si la VM no es estable, ir a (2) proxy residencial. Postergar 013-K hasta tomar esta decisión.

## Nota metodológica

Este spike NO escribe en SQLite, NO toca Notion y NO ejecuta yt-dlp con descarga de video — solo `extract_info(skip_download=True)`. Los datos crudos por video están en el JSON adjunto.
